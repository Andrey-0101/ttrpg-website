Мы завершаем текущий чат проекта и переходим к продолжению работы в новом чате.

Подготовь полноценный документ передачи контекста CURRENT\_HANDOFF.md. Добавь в начале названия порядковый номер формата HXXX, соответствующий позиции чата в общей последовательности. Документ должен позволить следующему чату продолжить разработку без повторного изучения всей переписки, без смешивания старых и новых версий кода и с минимальным риском ошибок, путаницы и галлюцинаций.

Проанализируй (из того, что действительно существует и относится к данному проекту):

•	весь текущий чат;

•	прикреплённые к проекту документы;

•	PROJECT\_CONTEXT.md, если он доступен;

•	тематические спецификации и ADR;

•	актуальный публичный GitHub-репозиторий;

•	только тот commit, который будет указан как текущий;

•	SQL migrations и generated database types, если они находятся в репозитории;

•	результаты сборки и проверок, предоставленные в этом чате.

Не считай старые сообщения или ранее предложенный код источником истины, если актуальный GitHub commit содержит более новую версию.

Используй следующий приоритет источников:

•	Код в указанном Git commit.

•	SQL migrations.

•	Generated database types.

•	Фактически проверенная конфигурация Supabase и Vercel.

•	CURRENT\_HANDOFF.md.

•	PROJECT\_CONTEXT.md.

•	Актуальные функциональные спецификации и ADR.

•	Текущий чат.

•	Старые чаты и архивные handoff-документы.

Перед генерацией документа проверь, достаточно ли у тебя фактической информации о текущем состоянии Git.

Мне необходимо предоставить или подтвердить:

•	repository URL;

•	active branch;

•	git status;

•	git rev-parse HEAD;

•	git rev-parse origin/main, если применимо;

•	git log -1 --stat --oneline;

•	результат npm run build;

•	был ли выполнен push;

•	был ли создан и проверен Vercel deployment;

•	есть ли незакоммиченные или untracked-файлы.

Если этих данных нет в текущем чате или они могли устареть, сначала выдай только список точных команд, которые мне нужно выполнить. Не создавай финальный handoff до получения результатов.

После получения необходимых данных создай CURRENT\_HANDOFF в формате md с учетом формирования названия, как было указано выше.

Для каждого существенного утверждения используй один из статусов:

•	VERIFIED — проверено по коду, сборке, базе, deployment или фактическому тесту;

•	IMPLEMENTED — реализовано в актуальном коде;

•	DECIDED — решение подтверждено, но может быть ещё не реализовано;

•	PLANNED — запланировано;

•	ASSUMED — предположение, которое необходимо проверить;

•	UNVERIFIED — информация существует, но не была подтверждена;

•	BLOCKED — дальнейшая работа зависит от выполнения условия;

•	DEPRECATED — старый подход, который больше нельзя использовать.

Строго отделяй:

•	существующее состояние от целевой архитектуры;

•	реализованное от запланированного;

•	проверенные факты от предположений;

•	актуальный код от устаревших вариантов из чата;

•	пользовательские требования от твоих рекомендаций;

•	временные решения от окончательных архитектурных решений.

Если чего-то нет или это не доступно, не додумывай, как могло бы быть, и не придумывай.

Не придумывай также:

•	таблицы;

•	колонки;

•	database constraints;

•	RLS policies;

•	Storage buckets;

•	Storage policies;

•	переменные окружения;

•	настройки Supabase Dashboard;

•	уже выполненные migrations;

•	выполненные тесты;

•	deployment status.

Если информация отсутствует, прямо отметь её как UNVERIFIED, ASSUMED или BLOCKED.

25\. Handoff Quality Check

Перед выдачей результата самостоятельно проверь:

•	нет ли противоречия между commit и описанным состоянием;

•	не указано ли запланированное как реализованное;

•	не потеряны ли принятые архитектурные решения;

•	не включены ли устаревшие версии кода;

•	отмечены ли непроверенные database и deployment assumptions;

•	указан ли один точный следующий шаг;

•	достаточно ли документа для продолжения без чтения полного чата;

•	нет ли secrets или данных из .env.



Повторяю: если в рамках этого чата чего-то нет или не известно, отмечай это в соответствии с инструкциями в данном сообщении. Не додумывай, не придумывай.

Выведи результат как один готовый Markdown-документ CURRENT\_HANDOFF.md.

Не добавляй перед ним длинное объяснение. После документа отдельно добавь только:

•	список рекомендуемых обновлений постоянной документации;

•	готовое opening message для нового чата.

Не включай secrets, API keys, passwords, service-role keys, .env values или персональные токены.

Не копируй в handoff полные исходные файлы, если они уже сохранены в GitHub. Используй пути, названия функций, краткие контракты и точный commit.

Если важный код существует только в текущем чате и ещё не сохранён в репозитории, явно обозначь это как UNCOMMITTED и включи достаточную информацию, чтобы его можно было восстановить.

Документ должен содержать необходимые метаданные, позволяющие тебе самому в дальнейшем определить и понять положение в последовательности относительно других, более ранних или более поздних чатов.
