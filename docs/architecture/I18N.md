# Internationalization

## Supported locales

```text
en
ru
```

Default locale:

```text
en
```

Application routes use an explicit locale prefix.

Examples:

```text
/en/characters
/ru/characters
```

## Routing

Configuration lives in:

```text
i18n/routing.ts
```

Current behavior:

- `localePrefix: "always"`;
- English and Russian are supported;
- the selected locale is retained in a cookie for one year;
- locale-less application URLs resolve through the localization middleware;
- application navigation should use helpers from `i18n/navigation.ts`.

Do not construct localized paths by manually concatenating `/en` or `/ru`.

## Request message loading

Configuration lives in:

```text
i18n/request.ts
```

The current request loader merges:

```text
messages/<locale>.json
messages/<locale>/vtm-v5.json
```

The system-specific VtM dictionary is separated from the common application dictionary.

## Current message files

```text
messages/en.json
messages/ru.json
messages/en/vtm-v5.json
messages/ru/vtm-v5.json
```

## Route locale and sheet language

These are separate concepts.

### Route locale

Controls:

- navigation;
- account and authentication UI;
- common buttons;
- route metadata;
- application errors;
- current VtM labels.

### Sheet language

Planned but not implemented.

The long-term requirement is that a character sheet can retain its own language independently of the current site route. No database field or selector currently exists.

Before implementation, decide:

- where the value is stored;
- default behavior for existing records;
- whether it can be changed after creation;
- whether print/export follows sheet language;
- how game-system dictionaries are loaded independently of route locale.

See the independent sheet-language ADR.

## Technical auth route

The email confirmation callback is:

```text
/auth/confirm
```

It remains outside `[locale]`.

The registration flow supplies a validated intended locale so the callback can redirect to a valid localized page.

Do not move the route or change proxy matching without retesting confirmation in both locales.

## Metadata and not-found behavior

The localized layout provides locale-aware metadata.

The application includes localized not-found behavior through:

```text
app/[locale]/not-found.tsx
app/[locale]/[...rest]/page.tsx
```

When adding a route:

- add localized visible strings;
- add route metadata when applicable;
- test both locales;
- verify no duplicate locale prefix;
- verify missing routes display the localized not-found UI.

## Error localization

Raw backend errors must not be shown to users.

Authentication errors are mapped to safe translation keys. New backend-facing features must follow the same rule.

## Terminology

VtM terminology is stored in the per-system dictionaries.

Known current Russian terminology includes:

```text
Опоры и Убеждения
Мощь Крови
Всплеск Крови
Объем исцеления
Бонус Дисциплин
Переброс Пробуждения Крови
Штраф кормления
Сила Кланового Проклятия
```

Known cleanup item:

- older standalone translation keys may retain earlier terminology even if the current combined UI no longer uses them.

## Future content architecture

UI strings belong in locale dictionaries.

Long-form game-hub content should use structured content files such as Markdown/MDX or another reviewed content system. Large articles should not be embedded in UI translation JSON.

A future game system should have its own dictionary or content namespace.

## Review checklist

For every localized change:

1. compare English and Russian key structure;
2. test both routes;
3. test long Russian labels at mobile width;
4. test metadata;
5. test error fallback;
6. run the production build.

A dedicated nested-key parity check is planned but not currently part of CI.
